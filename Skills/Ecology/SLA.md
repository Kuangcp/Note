# SLA

Service Level Agreement

1. 可用性 (Availability) —— "能不能用"
这是最直观的维度，通常用“几个 9”来衡量（如 99.9%）。

    计算公式: (1 - 服务不可用时间 / 总时间) * 100%。
    关键指标:
        MTBF (Mean Time Between Failures): 平均故障间隔时间，体现稳定性。
        MTTR (Mean Time To Repair): 平均修复时间，体现故障恢复能力。

2. 延迟/响应时间 (Latency) —— "快不快"
服务处理请求所需的时间。

    关键指标:
        平均响应时间: 容易被长尾请求掩盖问题。
        分位数 (P50, P90, P99): 比如 P99 <= 500ms，意味着 99% 的请求都在 500ms 内完成。这是评估用户体验最关键的指标。

3. 吞吐量 (Throughput/Traffic) —— "能承载多少"
服务在单位时间内能处理的请求量。

    关键指标:
        QPS/TPS: 每秒查询/事务数。
        并发数: 系统同时处理的请求数量。

4. 错误率 (Error Rate) —— "对不对"
请求失败占总请求的比例。

    关键指标:
        HTTP 5xx 占比: 服务器内部错误。
        业务逻辑错误率: 虽然接口通了，但返回了非预期的结果。

5. 饱和度 (Saturation) —— "还能撑多久"
衡量系统资源的使用情况（CPU、内存、磁盘 I/O、数据库连接池）。

    核心逻辑: 资源达到 100% 之前，性能往往就会开始下降。

6. 数据完整性/一致性 (Data Integrity)
对于数据库或存储类服务，SLA 还会包含：

    RPO (Recovery Point Objective): 最多允许丢失多久的数据。
    RTO (Recovery Time Objective): 故障后多久能恢复数据访问。
